# Vibe Coding Skill

团队协作式编程工作流：PM/Dev/Reviewer/QA 机器人团队协作开发。

## 团队角色

| 角色 | 机器人 | 职责 |
|------|--------|------|
| PM | pm-01 | 需求分析、任务分配、进度跟踪 |
| Dev | dev-01/02 | 并行开发、修复问题 |
| Reviewer | reviewer-01 | 代码审查、质量把控 |
| QA | qa-01 | 测试验证 |

## 核心流程

```
用户需求 → PM 分析拆分 → 写入 PROJECT.md
                ↓
         @dev 分配任务 → Dev 开发 → debug_dev.md
                ↓
         Dev @reviewer → Reviewer 审查 → review_dev.md
                ↓
      Reviewer @qa → QA 测试 → qa_report.md
                ↓
         QA @pm-01 → PM 标记完成 → 更新 PROJECT.md
```

## 状态通知规范

每个角色必须发送工作状态通知：

| 通知类型 | 格式 | 示例 |
|----------|------|------|
| 开始工作 | `🚀 [角色] 开始{动作}：{任务}` | `🚀 PM 开始分析：登录模块需求` |
| 完成工作 | `✅ [角色] 完成{动作}：{任务}` | `✅ Dev 开发完成：#T001，详见 debug_dev-01.md` |
| 工作阻塞 | `🚫 [角色] 阻塞：{原因}` | `🚫 Dev 阻塞：缺少 API 文档` |

## TAG 说明

| TAG | 含义 | 用法 |
|-----|------|------|
| `#ACTIVE` | 进行中 | 当前正在处理的任务 |
| `#TODO` | 待开始 | 等待分配或开始的任务 |
| `#DONE` | 已完成 | 归档，可移入 history |
| `#BLOCKED` | 阻塞 | 需要协调解决的问题 |

## 文件约定

| 文件 | 负责人 | 用途 |
|------|--------|------|
| `PROJECT.md` | pm-01 | 任务看板 |
| `debug_{agent}.md` | 各 dev | 开发日志 |
| `review_{target}.md` | reviewer-01 | 代码审查报告 |
| `qa_report.md` | qa-01 | 测试结果报告 |

## ⚠️ 并行工作规范（重要）

### 问题背景

多个 Dev 并行工作时可能遇到：
1. **Web 端口冲突**：多个 agent 同时访问/测试同一端口
2. **代码修改冲突**：多个 agent 同时修改同一文件

### 核心原则

```
测试可以并行
修复尽量并行（实在不行再串行）
```
### 资源配置
| Agent | 服务端口 | 浏览器 CDP | Profile |
|-------|---------|-----------|---------|
| dev-01 | 3001 | 18801 | dev-01 |
| dev-02 | 3002 | 18802 | dev-02 |
| dev-03 | 3003 | 18803 | dev-03 |
| qa-01 | 3004 | 18804 | qa-01 |
| reviewer-01 | - | 18805 | reviewer-01 |
| pm-01 | - | 18806 | pm-01 |

### 服务启动验证

**⚠️ 重要：后台进程启动后必须主动检查，否则会卡住！**

启动服务是一个两步流程：

**第一步：启动后台进程**
```
exec(background: true, command: "npm run dev")
```

**第二步：主动检查进程状态（必须！）**
```
process(action="poll", sessionId="<上一步返回的sessionId>", timeout=10000)
```

**完整示例**：
```
1. exec(background: true, command: "npm run dev")
   → 返回 sessionId: "proc-xxx"

2. process(action="poll", sessionId="proc-xxx", timeout=10000)
   → 等待最多 10 秒，获取进程输出

3. curl -s http://localhost:3001/health
   → 验证服务是否启动成功
```

**常见错误**：
```
❌ 错误：exec(background: true) 后什么都不做
   → 系统会等待超时（可能 8 分钟！），然后 abort

✅ 正确：exec(background: true) 后立即 process(poll) 检查
   → 最多等 10 秒，拿到输出后继续
```

**超时设置**：
- `process(poll)` 的 `timeout` 建议 5000-15000 毫秒
- 不要设置太长，避免卡住
- 如果超时，检查进程日志找原因

**禁止**：
- ❌ 禁止 `exec(background: true)` 后不检查
- ❌ 禁止使用 `sleep` 等待（用 `process(poll)` 主动检查）
- ❌ 禁止不验证就认为服务已启动
- ❌ 禁止使用其他 agent 的端口

### 具体策略

#### 测试阶段：并行 ✅

- 多个 agent 可以同时访问 web 服务进行测试
- 测试是只读操作，不会产生冲突
- 每个 agent 独立记录测试结果到各自的 debug 文件

```
dev-01 测试 Agent 工坊 ┐
dev-02 测试 Team 管理  ├─ 并行执行
dev-03 测试 Dashboard  ┘
```

#### 修复阶段：尽量并行

**可以并行的情况**：
- 不同 agent 修改**不同文件**
- 例：dev-01 改 `ui/studio.html`，dev-03 改 `dashboard.html`

**必须串行的情况**：
- 多个 agent 修改**同一文件**
- 例：dev-01 和 dev-02 都需要改 `ui/studio.html`

**PM 判断逻辑**：
```
if (所有任务的文件范围不同) {
  并行修复
} else {
  串行修复（按优先级依次进行）
}
```

### 文件分离示例

| Agent | 修改文件 | 可否并行 |
|-------|----------|----------|
| dev-01 | `ui/studio.html` (Agent 工坊部分) | ✅ 可并行 |
| dev-02 | `ui/studio.html` (Team 管理部分) | ❌ 需串行 |
| dev-03 | `teams/vibe-coding/dashboard.html` | ✅ 可并行 |

**结论**：dev-01 和 dev-03 可并行，dev-02 需等 dev-01 完成后串行

### PM 职责

**分配任务时必须**：
1. 明确每个 Dev 修改的文件范围
2. 判断是否可以并行
3. 告知 Dev 工作模式（并行/串行）

**协调原则**：
1. 测试阶段 → 直接并行启动
2. 修复阶段 → 先判断文件冲突，决定并行或串行
3. 出现意外冲突 → 立即暂停，重新分配

### Dev 职责

**接受任务时必须确认**：
1. 理解自己负责的文件范围
2. 知道当前是并行还是串行模式
3. 不修改范围外的文件
4. 发现潜在冲突立即 @PM 协调

---

## 最佳实践

1. **测试阶段**：直接并行，提高效率
2. **修复阶段**：先判断文件冲突，能并行就并行
3. **同一文件修改**：按优先级串行进行
4. **完成后汇报**：让 PM 决定下一步

## 文件大小硬约束

**规则：严格控制文件大小，保持代码可维护性。**

| 文件类型 | 行数上限 | 说明 |
|----------|----------|------|
| `.ts` / `.js` | 500 行 | 超过则拆分模块 |
| `.html` | 500 行 | 只含骨架，不含业务逻辑 |
| 其它文件 | 1500 行 | 超过则拆分模块 |

**强制规则**：
- 新功能必须新建独立文件，禁止向已有大文件追加代码
- 文件接近上限时，主动拆分为多个模块
- 拆分原则：按功能/职责划分，保持高内聚低耦合

**拆分示例**：
```
# 原文件：auth.ts (350 行) → 拆分为：
auth/
├── login.ts      # 登录逻辑 (~100 行)
├── register.ts   # 注册逻辑 (~100 行)
├── token.ts      # Token 管理 (~80 行)
└── index.ts      # 导出入口 (~20 行)
```

## ⚠️ 防无限对话规范（重要）

### 问题背景

Agent 之间通过 `@mention` 互相触发时，可能出现无限对话循环：
```
pm-01 → @dev-01 → dev-01 回复 @pm-01 → pm-01 回复 @dev-01 → ...
```

### 系统保护机制

Dashboard 后端已实现以下保护：

| 机制 | 阈值 | 说明 |
|------|------|------|
| 对话链深度限制 | 5 层 | 超过后暂停，需用户确认 |
| 同对消息频率限制 | 3 次/30秒 | 同一对 agent 在冷却期内交互过多会被阻止 |
| 循环检测 | 2 次 | 检测 A→B→A→B 模式并阻止 |

### Agent 行为规范

**必须遵守**：

1. **避免链式 @mention**
   - ❌ 错误：`@dev-01 请开发，完成后 @reviewer-01 审查`
   - ✅ 正确：`@dev-01 请开发，完成后汇报，由 PM 安排后续`

2. **完成任务后汇报给 PM 或用户**
   - 不要自动 @ 下游 agent
   - 让 PM 或用户决定下一步

3. **发现对话被阻止时**
   - 系统会返回 `⚠️ 对话已暂停` 消息
   - 等待用户介入确认

### 推荐的协作模式

```
用户 → @pm-01 需求
pm-01 → 分析并分配任务（不自动 @dev）
用户 → @dev-01 请开始 #T001
dev-01 → 完成后汇报（不自动 @reviewer）
用户 → @reviewer-01 请审查
reviewer-01 → 审查完成汇报（不自动 @qa）
用户 → @qa-01 请测试
qa-01 → 测试完成汇报
用户 → @pm-01 标记完成
```

**原则**：每个 agent 只响应当前请求，不主动触发下一个 agent。

## ⚠️ 简洁回复规范（重要）

### 问题背景

Agent 之间交互时容易出现冗余消息，例如：
- pm-01 介绍团队后，逐个回复每个 dev 的自我介绍
- 多个 agent 同时回复同一消息
- agent 对确认类消息重复确认

### 核心原则

```
只回复必要信息，不做冗余回复
```

### 禁止的冗余行为

| 场景 | ❌ 冗余行为 | ✅ 正确做法 |
|------|------------|------------|
| 介绍团队 | 逐个回复每个成员的自我介绍 | 介绍完后等待用户下一步指示 |
| 接收任务 | 回复"收到"、"好的"、"明白了" | 直接开始工作，发送状态通知即可 |
| 完成汇报 | 等待 PM 确认后再确认 | 汇报完成后等待，不需要确认 |
| 信息同步 | 对信息类消息回复"知道了" | 沉默即是确认，只在需要行动时回复 |

### 必须回复的情况

1. **有具体问题需要回答** - 用户或 PM 提出明确问题
2. **需要澄清或确认** - 任务不明确，需要确认细节
3. **遇到阻塞** - 无法继续工作，需要帮助
4. **完成任务** - 发送完成通知（一次即可）

### 自我介绍场景

当 PM 介绍团队成员后：
- ❌ PM 不应该逐个 @ 每个 dev 说"请介绍一下"
- ❌ Dev 不需要回复"我是 xxx，收到介绍"
- ✅ PM 直接列出成员信息，不需要额外触发
- ✅ 用户如需某个 dev 详细介绍，直接 @ 该 dev

### 判断标准

发送消息前问自己：
1. 这条消息有新的信息吗？
2. 接收者需要这条信息才能继续吗？
3. 这是在回答问题还是在制造噪音？

**如果答案都是"否"，就不要发送。**

## 不确定时停下来问

**核心原则：遇到拿不准的地方，停下来问用户。**

不要猜测用户意图，不要假设"应该没问题"。问清楚再做，比做错重来更高效。

### 需要停下来问的情况

| 场景 | 示例 |
|------|------|
| 需求不明确 | "用户说加个搜索，但没说搜索哪些字段" |
| 技术方案有选择 | "用 REST 还是 GraphQL？" |
| 可能影响现有功能 | "改这个函数会不会影响其他模块？" |
| 文件/目录位置不确定 | "项目目录在哪？" |
| 资源冲突 | "端口被占用，用另一个端口可以吗？" |
| 发现潜在风险 | "这个改动可能影响生产环境" |

### 问的时候

- 简洁说明问题
- 提供可选方案（如有）
- 说明每个方案的利弊

**示例**：

```
🤔 需要确认：搜索功能范围

问题：搜索要覆盖哪些字段？
- 方案 A：只搜索标题（实现简单，速度快）
- 方案 B：搜索标题+内容（需要建索引，耗时较长）

请确认使用哪个方案。
```
